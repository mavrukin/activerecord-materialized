# Changelog

## Unreleased

### Features

- Debezium `source.ts_ms` → source watermark, automatically. `ingest_debezium_change` now forwards a Debezium envelope's `source.ts_ms` (the DB commit time, monotonic within a source) as the `source_ts:` watermark — so a consumer relaying envelopes gets out-of-order suppression and `View.source_watermark` freshness (#106) for free, without hand-plumbing `source_ts`. Only an integer `source.ts_ms` is forwarded; an absent or non-integer value carries no watermark, so the change is maintained exactly as before. The top-level `ts_ms` (the connector's processing time) is intentionally *not* a fallback — it is a different clock, and mixing the two per partition would break the monotonicity suppression relies on. Builds on the Debezium adapter (#105) and source watermarks (#106). (#117)
- Source-watermark suppression now drops only a **strictly-older** change (`source_ts` `<` the partition's applied watermark), not an equal one. A coarse, second-granular timestamp — notably Debezium's MySQL `source.ts_ms` — can tie two *distinct* commits to a partition; treating an equal watermark as "already applied" silently dropped the second until reconciliation. An equal-`source_ts` change now recomputes (idempotent), so a distinct write is never lost; only an exact redelivery of the same watermark forgoes the skip-optimization. Surfaced by the #117 self-review. (#118)
- Optional CDC source watermarks (out-of-order suppression + freshness). `ingest_change` now accepts a monotonic `source_ts:` (a Debezium `ts_ms`, a per-key Kafka offset). The engine records the max applied watermark **per (view, partition)** in a new `ar_materialized_view_source_watermarks` table (lazily provisioned; added to the install migration), and skips a partition whose applied watermark is already newer than `source_ts` — an out-of-order change already superseded by a strictly-newer one is provably-stale, so its scoped recompute is suppressed. The watermark advances **at ingest**, so this is an optimization over always-recompute that is best-effort and **reconcile-backed** (a crash before the recompute can over-suppress a partition until reconciliation heals it), not exactly-once. A view's freshness is observable via `View.source_watermark` (the oldest applied partition watermark; subtract from your source clock for lag). Opt-in and backward-compatible: omit `source_ts` and behavior is unchanged; only the scoped-recompute path (`change_source :none`) carries a watermark, and a full recompute (no partition key) is never suppressed. See the [Change sources](README.md#change-sources) docs. (#106)
- Debezium CDC envelope adapter. `ActiveRecord::Materialized.ingest_debezium_change(envelope)` maps a decoded Debezium change envelope straight to the ingestion API — `op` c/r → create, u → update, d → destroy; the `before`/`after` row images; and `source.table` (unwrapping a nested `payload`) — so a consumer no longer hand-writes that boilerplate or its easy-to-miss cases: a snapshot read `r` is a create, a `nil` Kafka tombstone is a no-op, and a mis-shaped envelope (no `op` — e.g. a Maxwell event or a non-unwrapped message) or an unsupported op **raises** rather than being silently dropped. Pass a positional `table` to override the target when it differs from `source.table`. Pure and dependency-free (`DebeziumEnvelope.to_change_descriptor` is a hash→descriptor mapping — no Kafka/Debezium/Connect runtime, no deep copy of the row images). Correct partition-moving updates require full before-images (`binlog-row-image=FULL` / `REPLICA IDENTITY FULL`); with a primary-key-only before-image the old partition is under-maintained until reconciliation heals it. See the [Change sources](README.md#change-sources) docs. (#105)
- Real change-log CDC capture in the integration matrix. The literal end-to-end CDC path is now exercised: a raw write is **decoded from the database's own change log** and relayed through `ingest_change`, with no write-site descriptor synthesis — proving the ingestion API consumes exactly what a real log-based CDC consumer (Debezium/Maxwell-style) emits. A pluggable capture strategy per adapter (`spec/integration/support/cdc_capture.rb`, selected via `IntegrationAdapters::CAPTURE`): **Postgres** decodes a built-in `test_decoding` logical replication slot (`REPLICA IDENTITY FULL` so update/delete carry the partition key, not just the PK; slot lifecycle managed in setup/teardown); **MySQL** decodes ROW binlog events via `mysqlbinlog --read-from-remote-server -v`, mapping positional `@N` columns by ordinal (gated on the client — skipped with a logged reason when absent; CI installs it). Both decode to a normalized, Debezium-shaped descriptor (`op` c/u/d → create/update/destroy; name-keyed `before`/`after` full images). Deterministic — bounded reads, no `sleep`. SQLite has no server change log and is skipped. Documents the envelope→`ingest_change` mapping and the full-row-image requirement (`binlog-row-image=FULL` / `REPLICA IDENTITY FULL`). (#80)
- Trigger/outbox change source for out-of-band writes. A built-in, database-native way to capture writes that bypass **both** ActiveRecord callbacks and the ingestion API — raw SQL from a console, a bulk backfill, another service writing the shared database. `ActiveRecord::Materialized::WriteOutbox.install_triggers!(table, key_columns:)` installs `AFTER INSERT/UPDATE/DELETE` triggers that append the changed **GROUP BY key columns** (as JSON) to a durable outbox table, and `ActiveRecord::Materialized.drain_write_outbox!` relays them through `ingest_change` so the affected partition(s) are maintained exactly as for an ActiveRecord write. Capture is **scoped**: a create records the new-image keys, a destroy the old-image keys, and an update **both** — so a partition-moving update maintains the old partition and the new. The DDL is portable across the gem's adapters (Postgres: one trigger function branching on `TG_OP`; MySQL/SQLite: one trigger per operation) and emitted for the connection's adapter at migrate time via the `activerecord_materialized:outbox <table> <key_columns…>` generator. Draining is **at-least-once** (a redelivered row converges, since scoped recompute is idempotent), **batched** (bounded memory even for a large backfill backlog; `limit:` caps rows per pass), **per-row isolated** (one un-relayable row is left for retry and can't block the writes queued behind it), and a **no-op before the triggers are installed** (safe to schedule ahead of the migration). The outbox is a single change source like CDC — a fed view is `change_source :none`, and the triggers fire on every write (in-app included), so that one source covers all writes. This is the middle layer of the layered defense — between the ingestion API (hook it in code) and full CDC (a binlog/WAL pipeline) — with reconciliation remaining the time-bounded backstop underneath. Opt-in; not the default change source. Validated end-to-end against real MySQL, PostgreSQL, and SQLite. See the [detecting out-of-band writes](docs/out-of-band-writes.md) guide. (#68)
- Distribution-correct dispatch + reconcile fan-out. `refresh_dispatcher` now **auto-resolves to `:active_job` when ActiveJob is loaded** (otherwise the in-process `:async` thread), so a typical Rails deployment coordinates refresh across servers by default — an explicit `config.refresh_dispatcher` still wins, and a one-line boot warning fires when the single-process-only in-process dispatcher is active. New `ActiveRecord::Materialized.enqueue_stale_reconciles!` / `enqueue_stale_refreshes!` (and `materialized:enqueue_reconciles` / `materialized:enqueue_refreshes` rake tasks) fan the periodic bounded-staleness backstop out across the job fleet — **one job per stale view** — instead of one process verifying every view serially; a new `ReconcileJob` mirrors `RefreshJob` and re-checks staleness before working, so a duplicated tick is wasteful but never incorrect (and the additive summary-delta path is serialized by a row lock regardless). Run the tick from a single owner (leader / dedicated instance / one recurring job) — the gem ships no scheduler or leader election. `config.reconcile_queue_name` defaults to the refresh queue. See the [distributed deployment](docs/distributed-deployment.md) docs. (#93)
  - **Upgrade note:** an app with ActiveJob loaded that previously relied on the in-process `:async` default now dispatches refreshes as `RefreshJob`s. Ensure a worker drains `config.refresh_queue_name` (default `:materialized_views`), or set `config.refresh_dispatcher = :async` explicitly to keep the in-process refresher — otherwise refresh-on-write jobs enqueue with nothing to run them, and views serve stale reads. A boot warning fires when the in-process refresher is active despite ActiveJob being available.
  - **Cross-process cycle safety:** the non-idempotent summary-delta consume+apply is serialized by a metadata-row lock (#92), and the scoped-recompute path serializes its consume the same way (#95) — so concurrent cycles across servers apply an additive delta exactly once and recompute a partition once, rather than double-counting or duplicating rows (the latter previously possible on Postgres, which has no gap locks). The scoped apply runs outside the lock (it is idempotent), so a crash between a scoped consume and its apply loses that delta; self-healing reconciliation repairs it, so schedule `reconcile_stale!` as the backstop at scale.
  - **Writer/replica topology (#94):** `config.maintenance_role` / `config.verification_role` route maintenance (`refresh!`/`reconcile!`/`rebuild!`) to the primary and `DataVerifier` reads to a replica — the verify-on-replica / repair-on-primary split — via Rails `connected_to` (default `nil` = no routing, single-database apps unaffected). `DataVerifier` now reads the cache and the recomputed source in **one snapshot** (`REPEATABLE READ` where supported), so a write landing mid-verify — or replication lag — can no longer manufacture false drift. `config.replica_lag` folds a replication-lag budget into `stale?` so replica reads stay within `max_staleness`. Completes the epic; see the [distributed deployment](docs/distributed-deployment.md) guide.
- Self-healing reconciliation: a `Reconciler` + `ActiveRecord::Materialized.reconcile!` / `reconcile_stale!` API (and `materialized:reconcile` rake task) that bounds staleness in time — on a schedule it verifies each view (via the data-drift detector) and repairs whatever the change source missed, re-aggregating missing/mismatched partitions and dropping extra ones with **scoped** maintenance, never a full `rebuild!`. `reconcile_stale!` (the scheduled backstop) touches only dirty/never-refreshed/past-`max_staleness` views, so the missed-write→repair window is bounded by the schedule interval and each view's `max_staleness`. Safe alongside normal refresh: it drains pending maintenance first (so it repairs genuinely-missed writes) and repairs through the guarded, transactional maintenance path; when a refresh already holds the view's cycle the scoped repair is deferred (`ReconcileResult#deferred`) — durably queued and drained next, never lost — so it can neither corrupt the cache nor double-maintain. A clean reconcile resets the staleness clock (so a drift-free view isn't re-verified every tick), and a failure reconciling one view is isolated (`ReconcileResult#error`) rather than aborting a batch. Each run stamps `last_reconciled_at` / `reconciled_partition_count` on the metadata and emits a `reconcile.active_record_materialized` event. Builds on data-drift detection (#62) and composes with `max_staleness`; `cold_read :read_through` remains the always-correct fallback during repair. See the [Bounded staleness and self-healing](README.md#bounded-staleness-and-self-healing) docs.
- Observability via `ActiveSupport::Notifications`: the read, refresh, and maintenance lifecycle points now emit documented events (`read.active_record_materialized`, `refresh.active_record_materialized`, `maintenance.active_record_materialized`) with stable payloads — cache-hit vs read-through and staleness on reads; duration, mode, partitions recomputed and outcome on refreshes; and the maintenance path plus a **widen-to-full** signal on writes. No runtime dependency on a telemetry vendor; read events are emitted only when a subscriber is attached. See the [Observability](README.md#observability) docs.
- Data-drift detection: a `DataVerifier` + `ActiveRecord::Materialized.verify_data` / `verify_data!` API (and `materialized:audit` rake task) recompute the source relation and compare it to the cache per partition, reporting the divergent partition keys (`missing`/`extra`/`mismatched`) as a `DataVerificationResult`. Modes trade cost for depth — `:row_count` (partition presence), `:checksum` (per-partition value digest), `:full` (exact values). Comparison reads both the cache and the recomputed source through the cache model's own column types, so a value crossing two type systems (a date key, an integer/decimal column, a computed aggregate) isn't mistaken for drift, and duplicate cache rows are caught rather than collapsed; a schema-drifted or non-grouped view is skipped (an un-scaled decimal aggregate can still differ in trailing float precision — give it a scale). `sample:` (Integer count or Float fraction) is a cheap value-drift probe over materialized partitions for large views (it does not detect missing partitions; a full sample runs exhaustively). Covers grouped/aggregate views; stateless, like schema verification (#28); feeds self-healing reconciliation. See the [Detecting data drift](README.md#detecting-data-drift) docs.
- Join-keyed partition resolver: `partition_key_for(table) { |change| ... }` maps a write on a joined/leaf dependency table — whose own payload lacks the `GROUP BY` key — to the affected partition key(s), so maintenance stays **scoped** to those partitions instead of widening to a full recompute. The block returns a scalar or tuple (or arrays thereof; arity-aware); returning nothing falls back to a full recompute. Distinct from #47 (which qualified the predicate for already-known keys); this *derives* the keys. See the [joined-table docs](README.md#views-whose-group-key-lives-on-a-joined-table).
- CDC / external change-stream ingestion: `ActiveRecord::Materialized.ingest_change(table:, operation:, key_attributes:, before:, after:)` feeds a normalized change descriptor (from a Debezium/Maxwell-style stream, a bulk loader, or any external source) into the maintenance pipeline without an ActiveRecord object, building on the pluggable change-source seam. `WriteChange.from_descriptor` constructs a change from raw attributes. Delivery is forgiving: at-least-once and out-of-order converge (externally-fed views recompute), and key tuples are optional (absent → widen to a full recompute). No runtime dependency on a specific CDC tool. See the [Change sources](README.md#driving-maintenance-from-cdc--an-external-change-stream) docs.
- Pluggable change sources: change *detection* is now decoupled from view *maintenance* behind a documented public ingestion API — `ActiveRecord::Materialized.publish_write_change!` (a specific write) and `mark_dirty_for_tables!` (a coarse, idempotent full-recompute signal) — so writes can be fed from any source (bulk loads, raw SQL, other services, a CDC stream). Automatic commit-callback installation can be disabled globally (`config.default_change_source = :none`) or per view (`change_source :none`) while `depends_on` still declares dependency tables for scoping and metadata; the built-in callback tracker is implemented on top of the same ingestion API. Each view is fed by exactly one source (callbacks vs. ingestion API), and externally-fed views recompute affected partitions rather than applying signed deltas, so at-least-once/duplicate delivery converges. See the [Change sources](README.md#change-sources) docs.

## 0.1.0 (2026-06-18)

Initial release.

### Features

- Application-level materialized views for ActiveRecord (Rails 8+, Ruby 3.4+)
- Refresh-on-write: dependency changes schedule incremental background maintenance; reads never block on a rebuild
- Never an implicit full rebuild — a full materialization happens only via the explicit `rebuild!(confirm: true)` / `materialized:rebuild`, so launching against a large database is safe
- Read-through cold reads (`cold_read :read_through` default): reads on a not-yet-built view serve correct results from the source query; `:serve_stale` and `:raise` are also available
- Per-partition freshness: a cold view materializes individual `GROUP BY` partitions on demand (keyed reads and dependency writes), serving those partitions from the cache while the rest read through — partial materialization without ever a full rebuild
- Transparent ActiveRecord query interface (`where`, `find`, `count`, scopes)
- Declarative `materialized_from` sources defined as an `ActiveRecord::Relation` (via a block)
- `depends_on` dependency tracking via ActiveRecord `after_*_commit` callbacks
- Refresh strategies: `:async` (default), `:immediate`, `:manual`
- Debounced async refresh with in-process `AsyncRefresher` or ActiveJob dispatcher
- `rebuild!` materializes entirely in the database with `INSERT … SELECT` over the source query (atomic table swap), so the result set never crosses into Ruby memory — safe to run against a large dataset
- Warm-up: a `warm_up { [...] }` DSL plus `warm_up!` / `materialized:warm_up` materialize a cold view's hot partitions ahead of traffic, leaving the rest to read through on demand
- Incremental view maintenance (IVM) for `GROUP BY` views — never a routine table rebuild:
  - **Summary-delta** maintenance for distributive views (`SUM` / `COUNT` / `COUNT(*)`): writes apply signed per-partition deltas to the cache table without re-reading base rows, with NULL-safe sums and empty-partition deletion
  - **Scoped recompute** (partition-local delete + re-aggregate) for everything else — `AVG`, `MIN`, `MAX`, `COUNT(DISTINCT)`, joins, `HAVING` — always correct
- Metadata tracking (`dirty`, `last_refreshed_at`, `row_count`, `refresh_duration_ms`, errors)
- Optional `max_staleness` time-based safety net
- `before_refresh` / `after_refresh` callbacks
- Migration-provisioned cache tables: `activerecord_materialized:migration <View>` generates a `create_table` migration with columns/types inferred from the source relation, so the table exists at deploy time
- Boot/CI schema drift verification (`materialized:verify` / `ActiveRecord::Materialized.verify_schema!`) raises a helpful error when a view's table no longer matches its relation — never auto-alters
- Rails generators: `activerecord_materialized:install`, `activerecord_materialized:view`, `activerecord_materialized:migration`
- Rake tasks: `materialized:refresh_all`, `materialized:refresh_stale`, `materialized:rebuild`, `materialized:verify`, `materialized:warm_up`
- JOB-schema benchmark suite with multi-second analytical queries on SQLite

### Benchmark highlights (xlarge dataset, ~2M cast_info rows)

- Raw queries: 7–20 seconds
- Materialized view reads: ~0.3–0.7ms
- Speedup: 20,000–49,000×
